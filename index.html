Guided story

A narrative walkthrough of the tradeoff space. Adjust the controls on the left while reading — each claim updates in real time.

1

Start with the optimization goal

Before choosing a kernel path, set your goal. Speed, accuracy, portability, and simplicity all push the model in different directions. There is no configuration that maximizes all of them at once.

2

Choose a kernel path

The naive baseline keeps shared-memory and register pressure low, but pays for it with high global-memory traffic. Shared-memory tiling improves reuse by staging data on chip. The tensor-core MMA path raises arithmetic throughput sharply, but also increases implementation complexity and hardware dependency.

3

Watch pressure move across the system

As you tile more aggressively, global-memory traffic falls but shared-memory pressure rises. Adding double buffering hides latency but costs extra shared-memory budget. More register tiling keeps work local but can reduce occupancy because fewer warps fit on an SM.

4

Quantization adds its own axis

Lowering bitwidth reduces memory traffic substantially but increases the chance that clipping and group size choices affect numerical fidelity. Smaller groups help accuracy at the cost of scale overhead; larger groups do the opposite.

5

Interpret the tradeoff — don't chase every optimization

GPU optimization is not a checklist where every recommendation can be applied at once. Tiling, tensor-core execution, double buffering, scale caching, bitwidth, and group size each shift the balance between speed, occupancy, memory movement, accuracy, portability, and complexity.

Glossary

Key terms used in GPU kernel optimization, quantization, and the tradeoff simulator.

Kernel GEMM (General Matrix Multiply)
The core computational primitive in deep learning — multiplying two large matrices. Most of transformer inference time is GEMM time. GPU kernels implement GEMM using hierarchical tiling and parallelism across SMs, warps, and threads.
Kernel Shared-memory tiling
A technique where tiles of the input matrices are loaded from slow global DRAM into fast on-chip shared memory, then reused many times. Reduces global-memory traffic at the cost of on-chip shared-memory pressure.
Kernel Tensor-core MMA (Matrix Multiply-Accumulate)
Tensor cores are specialized hardware units on Ampere (SM80+) and later GPUs that execute small matrix operations in a single instruction. The MMA path uses warp-level primitives (wmma or ptx mma.sync) to feed them. Dramatically higher arithmetic throughput, but hardware-dependent and complex to implement.
Memory Occupancy
The ratio of active warps to the maximum number of warps an SM can hold. Higher occupancy means more warps are available to hide memory latency. Register pressure and shared-memory use are the main factors that reduce occupancy — they limit how many thread blocks can be resident simultaneously.
Memory Double buffering
A pipelining technique where two shared-memory buffers alternate: one is being consumed by compute while the other is being filled by the next DMA load. Hides memory latency at the cost of 2× shared-memory budget for buffered tiles.
Memory Register pressure
Each SM has a fixed register file shared across all resident threads. Kernels that use more registers per thread reduce the number of concurrent warps (lower occupancy). Register tiling — keeping accumulator tiles in registers — improves reuse but raises pressure.
Quant INT4 / Low-bit quantization
Representing weights in 4-bit integers instead of 16 or 32 bits. Reduces memory bandwidth and model size but introduces quantization error. In GEMM kernels, weights are dequantized on-the-fly using stored scale factors, which adds complexity and a small compute overhead.
Quant Group size
In group-wise quantization, each group of consecutive weights shares a single scale factor. Smaller groups (e.g. 32) give each scale less variation to cover, improving accuracy but increasing scale storage overhead. Larger groups (e.g. 128) are more memory efficient but can introduce more quantization error.
Quant Clipping / clip level
Before quantizing, outlier values can be clipped to a range. Aggressive clipping (lower clip level) reduces the dynamic range and can increase error for large activations. Conservative clipping (higher clip level) preserves outliers but gives less range to normal values.
HW SM (Streaming Multiprocessor)
The main compute unit of a GPU. Each SM contains warp schedulers, CUDA cores, tensor cores, shared memory, and a register file. Occupancy is measured per SM. SM80 refers to the compute capability of NVIDIA A100 — the minimum for tensor-core MMA with the wmma API.
HW Scale cache
A software-managed cache for dequantization scale factors. When scales are reused across multiple rows or columns, caching them avoids repeated loads from global memory. Reduces global traffic modestly but adds bookkeeping complexity.
HW Pipeline depth
How many software pipeline stages overlap memory movement with computation. Depth 0 = no overlap (sequential). Depth 1 = one prefetch stage. Depth 2 = two stages, maximizing overlap but requiring more shared memory and more complex control flow.

Use Cases

Common deployment and research scenarios, with recommended simulator settings for each.

LLM inference on datacenter GPU (A100 / H100)

Max throughput

Serving large transformer models where the bottleneck is memory bandwidth and arithmetic throughput. You want maximum tokens/sec. Tensor cores are essential. Double buffering hides HBM latency. INT4 or INT8 quantization with group size 64–128 is standard practice.

Kernel: Tensor-core MMA Goal: Max speed Bitwidth: 4–8 bit Double buffer: ON Tensor cores: ON Scale cache: ON

Edge / embedded inference (no tensor cores)

Portability

Deploying to GPUs without tensor-core support (older consumer cards, mobile GPUs) or cross-platform targets. Shared-memory tiling still gives good speedup over naive. INT4 saves bandwidth. Lower pipeline depth reduces complexity.

Kernel: Shared-memory tiled Goal: Portability Bitwidth: 4–8 bit Double buffer: OFF Tensor cores: OFF

Scientific / fine-tuning workloads (accuracy-critical)

Accuracy-first

Training or fine-tuning where numerical precision matters. You may still use BF16 or FP8 instead of INT4. If quantizing at all, use conservative clipping and small group sizes to keep error low. Occupancy can be sacrificed for precision.

Kernel: Tensor-core MMA Goal: Accuracy-first Bitwidth: 8–16 bit Group size: 32 Clip: 0.999+

Teaching / research prototype

Simplicity

Building a reference implementation to understand the tradeoffs, not to hit production throughput targets. The naive or tiled kernel is the right starting point — it's much easier to reason about and profile. Add complexity only when you understand the baseline.

Kernel: Naive baseline or Tiled Goal: Simplicity Pipeline depth: 0 Double buffer: OFF

Memory-constrained serving (small VRAM budget)

Memory efficiency

When VRAM is the binding constraint — fitting a larger model into a smaller GPU. INT4 quantization with a well-tuned group size is the primary lever. Scale caching helps reduce the bandwidth overhead of dequantization. Throughput is secondary to fitting the model.

Bitwidth: 4 bit Group size: 64–128 Scale cache: ON Goal: Balanced or Speed

References

Papers, documentation, and tutorials that ground the simulation model.

CUTLASS: CUDA Templates for Linear Algebra Subroutines — NVIDIA The primary reference for multi-level tiling strategy, warp hierarchy layout, and pipelined data movement in high-performance GEMM. The tile sizes, pipeline stages, and register/shared-memory partitioning in this simulator are directly inspired by CUTLASS design patterns. docs.nvidia.com/cutlass
Efficient GEMM in CUDA — Colfax Research Step-by-step tutorial explaining global memory coalescing, shared-memory bank conflicts, warp-level pipelining, and tensor-core programming. Used to calibrate the occupancy–tiling tradeoff relationships in this simulator. research.colfax-intl.com
GPTQ: Accurate Post-Training Quantization for Generative Pre-trained Transformers — Frantar et al., 2023 Foundational paper for INT4 quantization of transformer weight matrices. The group-size and clipping tradeoffs in this simulator reflect the accuracy/compression findings in GPTQ and its follow-on work. arxiv.org/abs/2210.17323
AWQ: Activation-aware Weight Quantization — Lin et al., 2023 Shows that a small fraction of weights are salient for accuracy, motivating per-channel scaling and careful clipping choices. Informs the clip-level control in this simulator. arxiv.org/abs/2306.00978
NVIDIA GPU Occupancy Calculator & Tuning Guide Official documentation on how register file size, shared-memory allocation, and thread block dimensions together determine SM occupancy. The occupancy model in this simulator follows the constraints documented here. docs.nvidia.com/cuda/cuda-occupancy-calculator
Roofline Model — Williams, Waterman, Patterson, 2009 The analytical model behind the pressure-bar visualizations. Arithmetic intensity and memory bandwidth together determine whether a kernel is compute-bound or memory-bound. The throughput vs. memory-efficiency axis in the radar chart is a direct application. dl.acm.org/doi/10.1145/1498765.1498785
FlashAttention / Tiled Attention Literature While focused on attention rather than GEMM, FlashAttention (Dao et al., 2022) popularized the idea of fused tiled kernels that keep data on-chip across operations — extending the same shared-memory reuse principles covered by this simulator to sequence-length-dependent workloads.
